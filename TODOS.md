### TODO-007: Structured Error Logging (Sentry)
**What:** Replace `console.error()` with Sentry or similar. Every error should have: user ID, request context, stack trace, and correlation ID.
**Why:** Vercel function logs are ephemeral and text-only. When something breaks for a paying customer, you need to find it in minutes, not hours.
**Pros:** Real-time error alerts. Error search. User-aware debugging. Free tier covers <5k errors/month.
**Cons:** One more service to manage. Sentry free tier might limit retention.
**Context:** `npm install @sentry/nextjs` → `npx @sentry/wizard@latest -i nextjs` → wrap API routes. Add `SENTRY_DSN` to env.
**Effort:** S (human: ~2hrs / CC: ~15min)
**Depends on:** Nothing

### TODO-004: Auth + Cross-Device History
**What:** Replace localStorage with Supabase-backed storage. Users log in with Google/GitHub, history persists across devices.
**Why:** localStorage clears when users clear browser data. Mobile users can't access desktop history.
**Pros:** Retention. Users come back because their history is there.
**Cons:** Adds auth complexity. JWT, sessions, RLS policies. Ship only when users ask for it.
**Context:** `ResumeVersion[]` schema is already defined and migration-ready — it maps directly to a Supabase table. Add Clerk or NextAuth for auth. Add `userId` to `ResumeVersion`.
**Effort:** M (human: ~1wk / CC: ~2hrs)
**Depends on:** User demand signal (multiple users complaining about lost history)

### TODO-009: Interview Prep Streaming
**What:** Convert `/api/interview-prep` on-the-fly generation to SSE streaming instead of a blocking JSON response.
**Why:** The current call blocks for 20-30 seconds with no loading feedback. User thinks it's broken.
**Pros:** Better UX. Matches the deep eval streaming pattern already in the codebase.
**Cons:** Requires changing the page client to consume SSE instead of `fetch().json()`. Moderate scope change.
**Context:** Pattern is already established in `app/api/evaluate/route.ts` + `app/evaluate/page.tsx`. Follow the same SSE structure. Move `max_tokens` to a streaming call using `anthropic.messages.stream()`.
**Effort:** S (human: ~3hrs / CC: ~20min)
**Depends on:** TODO-006 validated (users actually use interview prep)

### TODO-010: SSRF Allowlist for Custom Portals
**What:** Replace the hostname denylist in `app/api/scanner/portals/route.ts` with an explicit allowlist of known safe patterns, or disable redirect following.
**Why:** The current denylist (`localhost`, `10.x`, `192.168.x` etc.) can be bypassed via open redirects or cloud metadata IPs (169.254.169.254). An attacker who can add a custom portal URL can exfiltrate cloud credentials.
**Pros:** Eliminates SSRF class of attack at the portal input boundary.
**Cons:** Allowlisting is strict and may block legitimate company career pages on unusual hostnames. Better option: `{ redirect: 'error' }` in the Jina fetch call.
**Context:** The Jina fetcher in `lib/jina.ts` already runs any URL through Jina's proxy, which provides some isolation. The direct scan in `lib/scanner.ts` → `fetchJson()` does NOT proxy — it fetches directly from the serverless function. That's the real attack surface.
**Effort:** S (human: ~2hrs / CC: ~10min)
**Depends on:** Nothing
